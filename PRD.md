## 📊 一、市场调研与竞品分析

### 1.1 竞品概况

| 产品       | Star 数 | 核心定位       | 优势                         | 劣势                                        |
| ---------- | ------- | -------------- | ---------------------------- | ------------------------------------------- |
| **Walle**  | 12k+    | 企业级部署平台 | 功能完整、审核流程、灰度发布 | 配置复杂、学习成本高、PHP+Python 技术栈陈旧 |
| **Spug**   | 10.9k+  | 运维工作台     | 功能全面、主机管理           | 过于重型、不聚焦部署                        |
| **Goploy** | 活跃    | Web 部署系统   | Go 语言、性能好              | UI 欠佳、生态较小                           |

### 1.2 目标用户痛点（来自用户调研）

#### 🔴 高频痛点（P0）

1. **安全性缺失** - 没有身份认证，任何人都能部署，非常危险
2. **权限管理混乱** - 新手可以部署生产环境，导致事故频发
3. **无法溯源** - 出问题找不到是谁部署的，无法追责
4. **回滚困难** - 部署出错后不知道如何快速恢复
5. **通知缺失** - 部署失败了团队不知道，影响协作

#### 🟡 中频痛点（P1）

6. **配置繁琐** - 每次新建项目都要配置半小时
7. **日志不清晰** - 看不懂部署到哪一步了
8. **环境变量管理差** - 敏感信息明文存储
9. **多服务器部署慢** - 一台一台部署太耗时
10. **缺少审批流程** - 生产环境应该需要审核

#### 🟢 低频痛点（P2）

11. 没有部署统计和可视化
12. 不支持灰度发布
13. 缺少钉钉/企业微信集成

### 1.3 市场机会

- **Walle 的不足**: 配置复杂（初次使用需 1 小时），技术栈老旧（PHP+Python），部署繁琐（需 MySQL）
- **我们的机会**: 轻量级（5 分钟上手），现代化（Next.js 全栈），开箱即用（SQLite 单文件）

## 🎯 二、产品定位

### 2.1 产品名称

**DeployMaster** - 智能化轻量级部署平台

### 2.2 产品口号

> "5 分钟完成部署配置，让部署像发朋友圈一样简单"

### 2.3 目标用户

- **主要用户**: 5-50 人的中小型开发团队
- **次要用户**: 个人开发者、开源项目维护者
- **潜在用户**: 从 Walle 迁移过来追求简洁的团队

### 2.4 核心价值

1. **轻** - 比 Walle 轻 70%，单文件部署，无需 MySQL
2. **快** - 5 分钟完成配置，部署速度提升 3 倍
3. **安全** - 内置 JWT 认证、权限管理、操作审计
4. **智能** - AI 辅助配置、自动识别项目类型、智能推荐

## 🔐 六、安全性设计

### 6.1 安全策略

| 层级       | 防护措施       | 实现方案                       |
| ---------- | -------------- | ------------------------------ |
| **认证层** | 防止未授权访问 | JWT + RefreshToken，强制 HTTPS |
| **权限层** | 防止越权操作   | RBAC 权限模型，中间件校验      |
| **数据层** | 防止信息泄露   | 敏感信息 AES-256 加密          |
| **操作层** | 防止误操作     | 生产环境二次确认，审批流程     |
| **审计层** | 防止无法追溯   | 完整操作日志，不可篡改         |
| **网络层** | 防止攻击       | IP 白名单，速率限制，CSRF 防护 |

### 7.2 差异化竞争策略

#### 🎯 我们比 Walle 强在哪里？

1. **更轻量**: 单文件部署，不需要复杂的 PHP+MySQL 环境
2. **更现代**: Next.js 15 全栈，代码更易维护
3. **更智能**: AI 辅助配置，新手也能快速上手
4. **更快速**: WebSocket 实时推送，部署体验丝滑
5. **更简洁**: 4 级权限 vs 6 级权限，降低配置复杂度

#### 🎯 我们比 Spug 强在哪里？

1. **更聚焦**: 只做部署，不做"大而全"的运维平台
2. **更轻量**: 不需要 Agent，不需要 Redis
3. **更易用**: UI 更简洁，学习成本更低

## 🔧 十一、技术架构

### 11.1 技术栈选型

```
前端:
  - Framework: Next.js 15 (App Router)
  - UI: Ant Design / Shadcn UI
  - 状态管理: Zustand
  - 实时通信: Socket.io Client

后端:
  - Runtime: Node.js 20
  - Framework: Next.js API Routes
  - ORM: Prisma
  - 认证: NextAuth.js / 自研JWT
  - 任务队列: BullMQ + Redis

数据库:
  - 开发: SQLite
  - 生产: PostgreSQL 16
  - 缓存: Redis 7

部署:
  - 容器化: Docker + Docker Compose
  - CI/CD: GitHub Actions
  - 监控: Sentry (错误追踪)
```
